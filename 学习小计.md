## 2026-02-11 MySQL 建表踩坑

- 问题：从公众号复制代码导致 `\u00A0` 空格报错
- 解决：用 `cleanSQL()` 清理非法字符或者直接去github里面复制
- 教训：**永远不要直接粘贴富文本代码**，光哥里面用的是 不间断空格（No-break space）Unicode：\u00A0

### 代码示例

```js
function cleanSQL(str) {
  return str.replace(/\u00A0/g, ' ').trim();
}
```

## 2026-02-25 结构化输出

### 结构化输出的方式

1. JsonOutputParser (基础版)

原理：在 Prompt 中插入简单的 JSON 格式指令，拿到结果后直接 JSON.parse()。
特点：
    ❌ 无类型校验：只保证是 JSON，不保证字段对不对。
    ❌ 脆弱：LLM 多输出一句“好的，这是 JSON”，就会解析报错。
    ✅ 适用：极简场景，或你完全掌控 Prompt 且模型很听话时。
2. StructuredOutputParser (增强版)

原理：基于 Zod Schema 自动生成详细的 Prompt 指令，并在解析时进行类型校验。
特点：
    ✅ 强约束：通过 Prompt 强力引导模型输出特定结构。
    ✅ 类型安全：解析失败会抛出具体错误。
    ✅ 支持真流式：配合 stream() 可实现逐字/逐字段输出。
    ⚠️ 依赖模型配合：本质还是靠 Prompt 哄模型，偶尔会翻车。
    ✅ 适用：模型不支持 Tool Calls 时的首选方案，或需要真流式展示 JSON 时。
3. Tool Calls (原生能力)

原理：利用模型原生的 Function Calling 能力，将 Schema 定义为工具参数。
特点：
    ✅ 最可靠：模型经过微调，几乎 100% 按格式输出，无需 Prompt 哄骗。
    ✅ 语义清晰：明确区分“思考”和“行动”。
    ❌ 非流式友好：原生返回的是完整对象，无法逐字展示。
    ✅ 适用：后端数据处理、Agent 决策、对准确性要求极高的场景。
4. withStructuredOutput (智能路由 - 推荐默认使用)

原理：LangChain 的“大管家”。接收 Zod Schema，自动判断模型能力：
    优先：若模型支持 Native JSON Schema → 用最强约束模式。
    次选：若模型支持 Tool Calls → 伪装成工具调用。
    兜底：若都不支持 → 降级为 StructuredOutputParser。
特点：
    ✅ 开发体验最佳：一行代码搞定，不用关心底层。
    ⚠️ 流式局限：在前两种模式下，必须等完整 JSON 生成并校验通过后才返回（表现为“单一大 Chunk”），无法实现逐字打字机效果。
    ✅ 适用：90% 的非流式结构化需求。
5. JsonOutputToolsParser (流式神器)

原理：专为 Tool Calls + Streaming 设计的解析器。能在 token 流中实时拼接并检测完整的 JSON 参数。
特点：
    ✅ 实时响应：一旦模型生成完工具参数（哪怕后续话还没说完），立刻解析并触发工具调用。
    ✅ 用户体验：可实现“正在查询...”的即时反馈，无需等待模型说完所有话。
    ✅ 适用：需要流式展示 + 实时调用工具的高级场景。

## 2026-02-27 跑起来自动cli工具--mini-cursor(本质就是 stdin 读取)

1. npm run dev、pnpm install、vite、通常是在终端里跑。终端本身就是 shell。mini-cursor 是：程序内部调用子进程，这时候：直接绕过了终端。所以必须自己处理 shell 解析问题。

2. -- --template react-ts  第一个 -- 告诉 pnpm：后面的参数传给 create-vite，而不是传给 pnpm，告诉vite直接使用用户指定模板，就可以直接跳过交互问题。

## 2026-3-3 提示词模板prompt template

> 结合 Milvus 向量数据库 + LangChain 实战总结，聚焦 Few-Shot 语义示例选择

---

### 🧩 核心模板类型与用途

| 模板类型 | 作用 | 在本项目中的用途 | 关键特点 |
|--------|------|------------------|--------|
| **`PromptTemplate`** | 基础字符串模板，支持 `{变量}` 占位 | 构建周报主提示词，如 `“请根据以下内容写周报：{content}”` | 简单直接，适合单轮生成 |
| **`ChatPromptTemplate`** | 对话式模板（system/user/assistant 消息数组） | 让 LLM 扮演“周报助手”，统一语气和格式 | 更符合现代 LLM 的训练范式 |
| **`FewShotPromptTemplate`** | 自动生成带示例的完整提示词 | **核心组件**：注入从 Milvus 检索出的周报样例 | 显著提升输出风格一致性 |
| **`FewShotChatMessagePromptTemplate`** | Few-Shot 的对话形式（LangChain 中常通过组合实现） | 若需多轮对话+示例引导（如先展示例子再提问） | 融合角色设定与示例引导 |
| **`LengthBasedExampleSelector`** | 根据输入长度动态控制示例数量 | 防止提示词过长（如用户输入很长时，只选 2 个例子） | 控制 token 消耗，避免超限 |
| **`SemanticSimilarityExampleSelector`** | **已实现**<br>基于向量检索，选出语义最相近的示例 | 用户说“我们在做技术债”，自动匹配“重构+补测试”的周报样例 | **大幅提升输出相关性 ✅** |
| **`PipelinePromptTemplate`** | 组合多个子模板为一个完整 prompt | 拼接 “角色设定” + “动态示例” + “用户输入” | 模块化设计，易于维护和调试 |
